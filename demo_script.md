# TileLang-Mesh 毕业答辩系统演示剧本 (4 分钟版)

**目标**：由于硬件尚未流片，本次演示 100% 聚焦于“编译系统前端抽象与后端 TVM-IR 代码生成能力”，通过展示高层代码与底层复杂指令的强烈对比，证明编译优化（同步注入与调度）的核心工作量。

---

## 🎬 阶段一：前端高层语义展示（约 1 分钟）
**核心目标**：展示 TileLang-Mesh 写起来有多简单，体现“抽象收益”。

### 1. 实操动作
1. 打开 VS Code，开启**左右分屏**模式。
2. **左侧**：打开一个传统的 CUDA C++ 矩阵乘法文件（可从网上找一个标准的带有 `__syncthreads()` 的复杂实现）。
3. **右侧**：打开本项目中我们刚刚生成的统一演示脚本 `demo_presentation.py`。
4. 用鼠标高亮选中 `demo_presentation.py` 中 `matmul_persistent` 函数里的 `T.Pipelined` 循环以及网格张量 (`MeshTensor`) 的声明部分。

### 2. 演讲话术
> “各位老师，由于我们合作企业的 Sunmmio 分布式 NPU 目前仍处于流片前的验证阶段，因此今天的演示将聚焦于我们**编译系统的前端抽象与后端代码生成能力**。
> 大家看屏幕，左侧是传统 CUDA 的 GEMM 实现，开发者需要手动管理复杂的线程和共享内存。而右侧是我们用 TileLang-Mesh 编写的 GEMM。
> 开发者只需要用 Python 风格的高阶原语声明网格分布和流水线（鼠标指一下 `T.Pipelined`），完全不需要涉及底层的硬件同步逻辑。这极大地降低了分布式 NPU 的编程门槛。”

---

## 🎬 阶段二：编译器降级过程与依赖图剖析（约 2 分钟）
**核心目标**：展示编译器在背后默默做的硬核工作（构建依赖图与软流水线调度）。

### 1. 实操动作
1. 现场执行我们准备好的 FlashAttention 演示脚本（利用自动化流程一键生成编译分析结果）：
   ```bash
   ./demo_flashattn.sh
   ```
2. 打开 `pass_logs_flashattn` 文件夹，展示从 `00_Initial_Mod.py` 到底层数十个 Pass 生成的 IR 序列，体现编译器的“复杂展开能力”。
3. 打开刚生成的 **`dep_graph_flashattn.png` (依赖关系图)**。
4. 打开刚生成的 **`pipeline_flashattn.png` (软流水调度甘特图)**。

### 2. 演讲话术
> “现在，我们将极简的 FlashAttention 代码送入我们的编译器。
> 大家请看左侧生成的数十个 Pass 日志，原本极简的 Python 抽象，已经被我们的编译器按阶段展开成了底层 TVM-IR。
> （打开依赖图）请看这里，由于我们使用了 3 级流水线（`num_stages=3`），传统的静态分析无法处理这种跨越迭代的时序纠缠。编译器首先根据读写关系，成功**重构了底层的跨迭代依赖图（Dependency Graph）**。
> （打开甘特图）然后，我们的 B-level 启发式调度器，在满足依赖的前提下，将指令无缝地排布在张量核、向量核和 ODMA 上，实现了指令级的最大化并发，流水线几乎完全打满。”

## 🎬 阶段三：多版本物理缓冲分配与内存复用收益（约 1 分钟）
**核心目标**：用图表证明系统在内存生命周期维度的智能着色与极限压榨。

### 1. 实操动作
1. 接着打开刚生成的 **`buffer_flashattn.png` (内存生命周期图)**。
2. 运行我们专门编写的内存复用收益分析脚本（以表格形式量化成果）：
   ```bash
   python analyze_memory_reuse.py flashattn
   ```
3. 鼠标指向终端输出的汇总表格，以及图片中颜色的复用关系。

### 2. 演讲话术
> “为了隐藏访存延迟，高深度的流水线会导致大量局部变量的生命周期大幅度拉长且相互交叠。
> （打开内存图）请看大屏幕，这是编译器追踪出的所有变量生命周期。我们通过自研的**多版本物理缓冲分配算法**，将 28 个逻辑缓冲区，利用贪心图着色算法紧凑地映射到了 21 个物理槽位中。
> （展示终端表格）正如脚本分析显示，我们在极限压榨带宽、拉长变量存活期的前提下，依然精准地保证了空间的最大化利用。
> 这证明了我们的编译系统已经具备了将复杂大模型算子，高效、正确地降级并调度为分布式 NPU 底层指令的完整能力。演示完毕，谢谢大家！”

## 🛡️ 评委潜在提问“防翻车”话术

**Q：既然芯片没流片，你论文里第 5 章的那些加速比（46%）和流水线效率（100%）是怎么来的？**

**A (完美应对标准话术)：**
> “感谢专家的提问。确实，芯片目前尚未回片。
> 我们在第 5 章中所有的性能评估，都是**基于企业提供的 Palladium 硬件仿真器（Emulator）以及周期精确（Cycle-accurate）的代价模型**得出的。
> 我们在编译器后端生成了底层的 TVM-IR（就是刚刚大家看到的复杂依赖与指令），然后将其输入到硬件仿真器中，仿真器能够精确统计出指令发射的周期数、流水线气泡以及真实的 SRAM 占用情况。这在工业界芯片流片前期的**软硬件协同设计（Hardware-Software Co-design）**中，是标准且严谨的性能验证流程。”
