## 镜头 1：开场与高层抽象展示 (0:00 - 0:40)
**画面**：VS Code 左右分屏。左侧：复杂的 CUDA C++ GEMM 代码（需包含 `__syncthreads()` 等同步指令）；右侧：本项目 `demo_presentation.py` 中的 `matmul_persistent` 或 `flashattn` 函数。
**操作**：鼠标在左侧上下滚动展示复杂性，然后移动到右侧高亮 `T.Pipelined` 和网格张量声明。

**解说词**：
各位老师好，我是王昊泽，今天我将介绍集成了本文中张量编程模型与编译优化框架的ai编译器。
本次演示主要聚焦于**编译系统的前端抽象能力**和**后端优化效果**。
屏幕中展示了使用本文编程模型编写的NPU上的分布式矩阵乘算子。
与cuda代码不同，开发者只需通过 Python 风格的程序声明顶层的数据流，而无需陷入到硬件相关的细节中。这极大地降低了 NPU 这一特定AI加速器上的编程门槛。”
开发者使用本文中定义的网格张量也就是meshtensor声明张量到NPU多核上的映射关系。为了在切分后仍然保持数据的局部性，开发者为张量声明了本文中提到的blockwise zz layout。实际上布局对于开发者是一个相当底层的概念，编译器具备自动推断布局的功能。
在分布式矩阵乘中，开发者通过高层的allgather原语进行分布式系统上的数据交换，将不同核心上的数据收集到单个核心中用于后续计算。
至于维持编程模型正确性所需的同步控制流，由编译器自动注入。

接下来，我将对矩阵乘算子和flashattn算子进行编译，简单展示生成的tvmir。
可以看到，原本短短几十行的 Python 代码，已经被编译器降级为数百行的 TVM-IR。
编译器后端拥有几十个 Pass，它们负责一步步将高层语义降级为更加底层的指令，其中一部分由本文工作实现。
通过终端的输出可以看到，编译器在静态期精准地为代码自动注入了多达 143 处底层硬件同步逻辑。保证了并发操作的正确性。”


“不仅如此，系统还完成了极具挑战的软流水调度。
请看这张生成的依赖关系图。flashattention算子中指令间的数据依赖关系错综复杂，本文通过基于循环展开的跨迭代真依赖前向化和基于变量重命名的假依赖消除，将原始带环的数据依赖图严格转化为有向无环图，以释放并行潜力。
接着，基于这个依赖图，本文基于底层距离的启发式调度算法将指令排布在张量核、向量核和 ODMA 引擎上。
首先是矩阵乘算子的流水线甘特图，从中可以很清晰地看到流水线的特征：第一次计算所需的数据已经被预取完成，tensorcore可以直接开始计算，而本轮中也会为下一次计算准备数据。
这里则是flashattn流水线的甘特图。从中可以观察到一个有趣的现象：vectorcore被完全占满，其他两个引擎则相对闲置。这表明当前的算力瓶颈并不是通常意义上认为的tensorcore，而是vectorcore。在与合作企业沟通后，企业认为这一发现能够为下一代芯片研发提供了重要参考。


“最后，我们来看内存优化。
为了隐藏访存延迟，深度的流水线会让大量局部变量的生命周期大幅拉长并相互重叠。
基于上述调度结果，编译器会追踪出所有变量生命周期。通过本文多版本物理缓冲分配与贪心图着色算法，系统成功将 29 个逻辑缓冲区，极其紧凑地映射到了 20 个物理槽位中。
从终端的数据统计可以看出，我们在维持流水线性能不变前提下，依然节省了约 35% 的物理内存空间。
本编译器将作为合作企业自研npu a4e适配的编译器，力求让开发者准确且高效地表达分布式npu上的语义。
我的演示到此结束，感谢观看。”
